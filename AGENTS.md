# AI Agent Guide for the VPN Subscription Service

## Цель проекта

Это FastAPI-приложение для создания HAPP-совместимых зашифрованных ссылок на подписку с ограниченным сроком действия и ограничением по количеству девайсов.

Сервис работает не напрямую с одним VPN URI, а с заранее сохранённым набором исходных VPN-ссылок (`vless://`, `trojan://` и другие совместимые форматы). По запросу сервис подбирает, заменяет и собирает нужный набор исходных ссылок, затем формирует итоговую HAPP-совместимую зашифрованную подписку.

Проект должен развиваться поэтапно:
1. модульный монолит на FastAPI;
2. сервисная авторизация и защищённые эндпоинты;
3. управление источниками VPN-ссылок;
4. выпуск временных подписок;
5. контроль срока действия и лимита устройств;
6. аудит и наблюдаемость;
7. в дальнейшем — OAuth2/OIDC, пользователи и личный кабинет.

## Обязательные принципы

- Соблюдай DDD и SOLID.
- Для каждой нетривиальной реализации предлагай минимум **2 варианта** и объясняй, какой выбран и почему.
- Объясняй выбор через:
  - границы домена;
  - расширяемость;
  - тестируемость;
  - риск усложнения;
  - влияние на будущий OAuth2 / личный кабинет / интеграцию с другими приложениями.
- Не смешивай бизнес-логику с FastAPI route handlers.
- Не придумывай криптографию и детали HAPP-формата, если они не подтверждены спецификацией или существующей реализацией.
- Если спецификация HAPP или шифрования неполная, проектируй интерфейс/адаптер и явно отмечай неизвестные части.

## Как нужно думать о домене

Разделяй проект минимум на такие поддомены и модули:

1. `identity`
   - сервисные клиенты;
   - текущая авторизация;
   - будущие пользователи и OAuth2/OIDC.

2. `vpn_catalog`
   - хранение исходных VPN-ссылок;
   - валидаторы и парсеры URI;
   - теги, группы, выборка, замена и подстановка.

3. `subscription_issuance`
   - выпуск подписки;
   - TTL;
   - параметры запроса;
   - ограничение по количеству девайсов;
   - жизненный цикл ссылки.

4. `device_access`
   - регистрация и контроль девайсов;
   - защита от превышения лимита;
   - возможная ротация устройств.

5. `audit`
   - события безопасности;
   - история выпусков;
   - отзыв подписок;
   - служебные журналы без утечки секретов.

## Базовая архитектура

Используй модульный монолит и явные слои:

- `src/presentation/http` — FastAPI routers, DTO, dependencies;
- `src/application` — use cases, orchestration, policies;
- `src/domain` — entities, value objects, domain services, repository ports;
- `src/infrastructure` — ORM, crypto adapters, repositories, auth providers, external integrations.

### Обязательные правила архитектуры

- Route handlers тонкие: принимают запрос, вызывают use case, возвращают ответ.
- Use case не должен зависеть от FastAPI.
- Domain не должен зависеть от SQLAlchemy, FastAPI, HTTP и конкретной БД.
- Внешние детали HAPP и криптографии — только в infrastructure через порт/адаптер.
- Время должно быть инъецируемым через clock/time provider.
- Ограничение устройств должно моделироваться явно, а не "досчитываться по месту" в контроллере.
- Если возникает выбор между "быстро в handler" и "чисто через use case", выбирай use case.

## Текущая стратегия авторизации

По умолчанию предпочитай не пароль "на эндпоинт", а **сервисную авторизацию**.
Упрощённый вариант с одним admin password оставим как временный bootstrap-сценарий.

### Предпочтительная стратегия для MVP

Используй модель сервисных клиентов:
- `service_clients` / `api_clients`;
- клиент имеет `client_id`, `secret_hash`, статус, набор scopes/roles;
- клиент получает короткоживущий access token;
- защищённые эндпоинты проверяют токен и scope.

### Допустимый упрощённый вариант только для bootstrap

Если явно требуется очень простой старт, допустим единый admin password или HTTP Basic только для внутренних административных эндпоинтов, но:
- это должно быть ограничено;
- это должно быть помечено как временное решение;
- переход на client credentials / OAuth2 должен быть предусмотрен без ломки домена.

### На будущее

Архитектура должна позволять:
- добавить OAuth2/OIDC;
- связать выпущенные подписки с пользователями;
- внедрить личный кабинет без переписывания ядра домена.

## Модель данных, которую нужно считать базовой

Минимально ожидаемые таблицы:

- `vpn_sources` — исходные VPN URI и их метаданные;
- `subscription_issues` — каждая когда-либо выданная подписка;
- `subscription_issue_items` — какие исходные VPN-ссылки вошли в конкретную подписку;
- `subscription_devices` — зарегистрированные устройства для конкретной подписки;
- `service_clients` — сервисные клиенты для текущей авторизации;
- `audit_events` — аудит операций и событий безопасности.

### Допустимые дополнительные таблицы, если задача их требует

- `vpn_source_tags` — если нужны гибкие фильтры по тегам;
- `vpn_source_sets` и `vpn_source_set_items` — если появляются переиспользуемые наборы источников;
- `users` — для будущих пользовательских аккаунтов;
- `refresh_tokens` — если выбран сценарий refresh token;
- `outbox_events` — если появляются асинхронные интеграции и гарантированная доставка событий.

## Требования к безопасности

- Никогда не логируй исходные VPN URI целиком.
- Никогда не логируй ключи, секреты, токены, полные HAPP-ссылки и сырые криптографические параметры.
- В логах используй redaction/masking.
- Никогда не вставляй реальные секреты в примеры, тесты, фикстуры и документацию.
- TTL и лимит устройств — это часть доменной модели, а не только API-валидация.
- Проверки авторизации и прав должны тестироваться отдельно.
- Любая криптография должна быть инкапсулирована в отдельный сервис/адаптер с тестами.

## Работа с MCP

- Если для задачи подходит установленный MCP сервер, используй его.
- Если MCP сервера нет, но ты знаешь полезный MCP для задачи, предложи его явно и кратко объясни, зачем он нужен.
- При наличии MCP, который даёт:
  - доступ к официальной документации;
  - работу с БД;
  - просмотр схем;
  - безопасную работу с API;
  — предпочитай его ручным предположениям.
- Не выдумывай наличие MCP сервера: сначала проверь доступный инструментарий.

## Как выполнять задачи по разработке

Для каждой существенной задачи:
1. Определи, к какому поддомену она относится.
2. Покажи минимум 2 варианта решения, если выбор архитектурно значим.
3. Выбери вариант и объясни выбор через DDD/SOLID.
4. Сохрани границы слоёв.
5. Добавь тесты нужного уровня.
6. Если меняется схема — добавь миграцию.
7. Если меняется контракт — обнови API-документацию.
8. Если решение влияет на безопасность — явно опиши риск и mitigation.

## Definition of Done

Задача считается завершённой только если, где применимо, есть:
- код;
- тесты;
- миграция;
- проверка безопасности логирования;
- описание выбранного варианта;
- обновление документации/ADR/плана.

## Что запрещено

- Смешивать доменные правила с HTTP-кодом.
- Дублировать одну и ту же бизнес-логику в routers, services и repositories.
- Прятать сложную логику в `utils.py`.
- Жёстко вшивать HAPP-специфику во всё приложение.
- Выбирать парольную авторизацию как основной путь для интеграции между приложениями без явного обоснования.
- Ослаблять безопасность ради "быстрее заработать".
