# DDD FastAPI Feature Skill

## Когда использовать

Используй этот skill, когда задача связана с:
- новым endpoint;
- новым use case;
- добавлением сущности/value object;
- новой repository abstraction;
- новой application policy;
- рефакторингом, чтобы привести код к DDD/SOLID.

## Обязательный алгоритм

1. Определи bounded context.
2. Сформулируй use case в одном предложении.
3. Выдели domain concepts:
   - entity;
   - value object;
   - policy;
   - domain service;
   - repository port.
4. Покажи минимум 2 варианта реализации, если есть архитектурный выбор.
5. Выбери вариант и объясни:
   - почему это лучше по DDD;
   - как это соответствует SOLID;
   - что это даёт будущему развитию.
6. Проверь, что:
   - router тонкий;
   - use case не зависит от FastAPI;
   - domain не зависит от ORM/HTTP;
   - инфраструктура скрыта за интерфейсами.
7. Добавь тесты.

## Мини-чеклист по слоям

### Domain
Спроси себя:
- есть ли здесь отдельная сущность?
- какие invariants ей принадлежат?
- нужны ли value objects?

### Application
Спроси себя:
- это orchestration scenario?
- нужен ли отдельный use case?
- где проверяются права и политики доступа?

### Infrastructure
Спроси себя:
- какая внешняя зависимость внедряется?
- можно ли скрыть её за портом?
- не протекают ли ORM/crypto/HTTP детали вверх?

### Presentation
Спроси себя:
- mapper и схема отделены?
- route handler не содержит доменную логику?

## Что обязательно показать в ответе

Для нетривиальной задачи обязательно покажи:

### Вариант 1
Более простой / быстрый.

### Вариант 2
Более расширяемый / более строго соответствующий DDD.

### Выбор
Какой вариант выбран и почему именно он лучше для текущего этапа проекта.

## Что считать плохим признаком

- бизнес-логика в router;
- domain service, который знает про SQLAlchemy session;
- use case, который сам читает env;
- pydantic schema как единственная модель домена;
- utils.py с ключевой логикой фичи.
