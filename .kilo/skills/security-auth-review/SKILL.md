# Security Auth Review Skill

## Когда использовать

Этот skill нужен для задач:
- как защитить эндпоинты;
- как авторизовать другие приложения;
- как хранить client secrets;
- как перейти от простого пароля к OAuth2/OIDC;
- как не утечь секретами в логах;
- как спроектировать права доступа.

## Базовая рекомендация проекта

Для интеграции с другими приложениями по умолчанию рекомендуй:

### Вариант 1 — service clients + short-lived JWT
Предпочтительный путь для этого проекта.

Плюсы:
- лучше масштабируется;
- ближе к будущему OAuth2;
- позволяет scopes/roles;
- лучше подходит для machine-to-machine.

### Вариант 2 — единый пароль / HTTP Basic
Только как bootstrap-времянка.

Плюсы:
- быстро;
- минимальный порог входа.

Минусы:
- хуже аудит;
- хуже отзыв и ротация;
- неудобно масштабировать;
- плохо эволюционирует к полноценной auth модели.

## Обязательные проверки

При любой auth/security задаче проверь:
- где хранится секрет;
- хранится ли он в hash, а не в открытом виде;
- как выдаётся токен;
- как проверяются scopes/roles;
- какие эндпоинты admin/internal/public;
- какие отрицательные тесты есть;
- что попадает в логи.

## Device-limit и TTL

Если задача затрагивает выдачу подписок:
- проверь, что max_devices enforce-ится сервером;
- проверь, что TTL проверяется сервером;
- проверь, что отзыв подписки не полагается только на клиентскую честность.

## MCP

Если доступен подходящий MCP для:
- официальной документации auth библиотек;
- схемы БД;
- анализа OpenAPI;
- безопасной интеграции;
используй его.

Если MCP нет, но он помог бы, предложи его отдельно.
