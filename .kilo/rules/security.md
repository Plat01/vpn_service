# Security Rules

## Security-first

В этом проекте безопасность важнее удобства реализации.

Агент обязан проектировать и реализовывать решения так, как будто:
- исходные VPN URI чувствительны;
- итоговые HAPP-ссылки чувствительны;
- любое логирование может попасть в внешние системы;
- токены и client secrets могут быть скомпрометированы;
- лимит устройств и TTL — часть защищаемой модели доступа.

## Секреты и чувствительные данные

Нельзя логировать:
- полные `vless://`, `trojan://` и другие исходные URI;
- полные HAPP-ссылки;
- client secret;
- access token / refresh token;
- сырой криптографический материал;
- полный request body, если в нём есть VPN URI.

Нужно:
- редактировать/маскировать значения в логах;
- использовать хеши и fingerprints, когда нужна трассировка;
- делать безопасные примеры и фикстуры.

## Правила авторизации

До появления полноценного OAuth2/OIDC агент должен по умолчанию предпочитать один из двух вариантов:

### Вариант A — service clients + token endpoint
Предпочтительный вариант для интеграции с другими приложениями.

- хранить `client_id`;
- хранить только `secret_hash`;
- выдавать короткоживущий access token;
- использовать scopes/roles;
- явно разделять административные и прикладные эндпоинты.

### Вариант B — bootstrap admin password / HTTP Basic
Допустим только как временный этап.

- только для внутреннего административного контура;
- только с чёткой маркировкой как временное решение;
- только если структура кода не мешает перейти на вариант A.

Если нет жёсткого ограничения, агент должен рекомендовать вариант A.

## Криптография

- Не придумывать собственные алгоритмы.
- Не размазывать криптографию по коду приложения.
- Любая криптография должна быть инкапсулирована в адаптер/сервис.
- Конфигурация криптографии должна быть централизована.
- Проверки корректности сериализации/десериализации и шифрования должны покрываться тестами.

Если спецификация HAPP-шифрования неизвестна:
- реализовать порт `HappLinkEncoder` / аналогичный интерфейс;
- описать, какие части подтверждены, а какие заглушены;
- не выдавать неподтверждённую совместимость за факт.

## TTL и лимит устройств

- TTL и max_devices — доменные ограничения.
- Они должны храниться явно в модели подписки.
- Проверка не должна жить только в request schema.
- Факт первого использования, регистрация устройства, превышение лимита и отзыв подписки должны быть наблюдаемыми событиями.

## Хранение и аудит

- Храни отдельно исходные источники и выданные подписки.
- Храни журнал выдачи, отзыва, попыток использования и ошибок авторизации.
- Журнал должен быть полезным для расследования, но безопасным с точки зрения утечки секретов.

## Дополнительные меры

По умолчанию предлагай:
- rate limiting для auth и issuance endpoints;
- идемпотентность там, где возможны повторные запросы от приложений;
- request ID / correlation ID;
- безопасные заголовки;
- строгую обработку ошибок без утечки внутренних деталей;
- проверку срока действия на сервере, а не доверие клиенту.
