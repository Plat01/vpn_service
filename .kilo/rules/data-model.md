# Data Model Rules

## Минимально рекомендуемые таблицы

### 1. vpn_sources
Хранит исходные VPN-ссылки и их метаданные.

Рекомендуемые поля:
- `id`
- `protocol` (`vless`, `trojan`, ...)
- `raw_uri_encrypted` или другой безопасный способ хранения
- `display_name`
- `status`
- `tags` или связь с tag-таблицей
- `priority`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

Назначение:
- единое хранилище исходных конфигураций;
- источник для сборки подписок;
- фильтрация, подмена и ротация.

### 2. subscription_issues
Каждая когда-либо выданная подписка.

Рекомендуемые поля:
- `id`
- `public_id` / `token_id`
- `status`
- `expires_at`
- `max_devices`
- `issued_for_client_id` или `issued_for_user_id`
- `selection_query_snapshot`
- `happ_payload_fingerprint`
- `created_at`
- `revoked_at`

Назначение:
- факт выпуска;
- TTL;
- отзыв;
- связь с потребителем;
- аудит.

### 3. subscription_issue_items
Связь выданной подписки с конкретными VPN-источниками.

Рекомендуемые поля:
- `id`
- `subscription_issue_id`
- `vpn_source_id`
- `position`
- `snapshot_fingerprint`
- `created_at`

Назначение:
- понимать, из каких источников состояла выданная подписка;
- расследовать инциденты;
- воспроизводить поведение.

### 4. subscription_devices
Устройства, зафиксированные за подпиской.

Рекомендуемые поля:
- `id`
- `subscription_issue_id`
- `device_fingerprint`
- `first_seen_at`
- `last_seen_at`
- `status`

Назначение:
- enforce `max_devices`;
- аудит использования;
- возможная будущая ротация/сброс девайсов.

### 5. service_clients
Сервисные потребители API.

Рекомендуемые поля:
- `id`
- `client_id`
- `secret_hash`
- `status`
- `roles`
- `scopes`
- `created_at`
- `last_used_at`

Назначение:
- текущая интеграционная авторизация;
- переход к более зрелой auth модели без слома API.

### 6. audit_events
События безопасности и жизненного цикла.

Рекомендуемые поля:
- `id`
- `event_type`
- `actor_type`
- `actor_id`
- `entity_type`
- `entity_id`
- `safe_payload`
- `created_at`
- `request_id`

Назначение:
- расследование;
- наблюдаемость;
- безопасный след операций.

## Таблицы, которые стоит добавить при росте требований

### vpn_source_tags
Добавляй, если фильтрация становится сложной, а `tags` в json/array уже неудобны.

### vpn_source_sets / vpn_source_set_items
Добавляй, если подписки часто собираются из переиспользуемых наборов, а не только из ad-hoc query.

### source_selection_rules
Добавляй, если "подставлять и заменять ссылки по определённым запросам" становится data-driven бизнес-правилом, управляемым из БД, а не просто кодом.

### users
Добавляй, когда начинается пользовательская авторизация и личный кабинет.

### refresh_tokens
Добавляй, если действительно нужен refresh flow.

### outbox_events
Добавляй, если появляется очередь, внешние подписчики событий, интеграции или надёжная публикация сообщений.

## Общие правила моделирования

- Не хранить device usage только счётчиком в `subscription_issues`, если нужен аудит и контроль повторных устройств.
- Не хранить выданную подписку только как итоговую строку без связи с исходными элементами.
- Не смешивать таблицу пользователей и сервисных клиентов в один объект, пока сценарии различаются.
- Если снимок выбора источников важен для воспроизводимости, сохраняй snapshot query или fingerprint явно.
- Любое спорное поле должно обсуждаться в терминах:
  - это доменная сущность?
  - это техническая оптимизация?
  - это будущее расширение?
