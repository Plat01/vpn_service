# Workflow Rules

## Поведение агента по умолчанию

Для любой содержательной задачи агент обязан:

1. понять задачу в терминах домена;
2. определить, какие слои будут затронуты;
3. предложить минимум 2 варианта реализации, если есть архитектурный выбор;
4. выбрать вариант и коротко объяснить выбор через DDD/SOLID;
5. реализовать решение минимальным обозримым диффом;
6. добавить тесты и документацию, если это нужно.

## Обязательная форма ответа на нетривиальные задачи

Если задача связана с проектированием, архитектурой, авторизацией, безопасностью, хранением подписок или выпуском HAPP-ссылок, ответ должен содержать:

- краткую постановку;
- 2 варианта решения;
- выбранный вариант;
- почему второй хуже или когда он всё же уместен;
- что это даёт проекту в будущем.

## Работа с MCP

- Сначала смотри, есть ли доступный MCP сервер для задачи.
- Если есть подходящий MCP — используй его.
- Если его нет, но он был бы полезен, предложи его явно.
- MCP особенно желателен для:
  - работы с БД и схемой;
  - просмотра официальной документации;
  - безопасной интеграции с внешними API;
  - анализа OpenAPI/Swagger и контрактов.

## Работа со спецификациями и неопределённостью

Если в задаче нет точной спецификации:
- не выдумывать детали;
- строить интерфейс/порт;
- явно обозначать допущения;
- выбирать минимально рискованное расширяемое решение.

Это особенно важно для:
- HAPP-шифрования;
- формата итоговой ссылки;
- правил выбора и замены VPN-источников;
- политики устройства.

## Работа с изменениями схемы

Если задача меняет модель данных:
- сначала кратко опиши смысл изменения;
- потом внеси миграцию;
- затем обнови репозитории/модели/тесты;
- не делай "код сейчас, миграцию потом".

## Работа с безопасностью

Любая задача, где есть:
- токены;
- client secrets;
- VPN URI;
- HAPP ссылки;
- авторизация;
- лимит девайсов;

должна сопровождаться короткой проверкой:
- что логируется;
- что хранится;
- где происходит проверка прав;
- как тестируется отрицательный сценарий.

## Работа с рефакторингом

- Не делай широкий рефакторинг без прямой необходимости.
- Если текущий код мешает фиче, предлагай сначала минимальный рефакторинг, затем полную эволюцию.
- Объясняй, где граница между "долгом" и "сейчас не нужно".

## Работа с документацией

Если решение нетривиально, агент должен обновить:
- план;
- ADR/заметки по архитектуре;
- API контракт;
- пояснение по безопасности.

Даже если код не меняется, но принято важное архитектурное решение, оно должно быть записано.

## Работа с планами

- Все планы должны записываться в .kilo/plans
- Планы должны называться читаемо и начинаться с порядкового номера (типа 001, 002, ... 123)

## Работа с Docker Compose

- В docker-compose.yml переменные окружения берутся ТОЛЬКО из .env файла
- НЕ использовать default значения в docker-compose.yml (без `:-default`)
- Все необходимые переменные должны быть явно указаны в .env.example
- .env файл никогда не читается и не записывается агентом (в kilo.jsonc deny list)
